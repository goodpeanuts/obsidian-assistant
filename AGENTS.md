# Repository Guidelines

## Project Structure & Module Organization
This repository is an Obsidian plugin written in TypeScript.

- `main.ts`: plugin entrypoint, command registration, and module wiring.
- `settings.ts`: settings model, defaults, and settings tab UI.
- `core/`: archive logic (`scanner.ts`, `time-resolver.ts`, `archiver.ts`).
- `ui/`: modal and user-facing interaction (`archive-modal.ts`).
- `manifest.json`: plugin metadata consumed by Obsidian.
- `main.js`: bundled output generated by esbuild (do not hand-edit).

Keep new domain logic in `core/` and keep Obsidian UI concerns in `ui/`.

## Build, Test, and Development Commands
- `npm install`: install dependencies.
- `npm run dev`: start esbuild watch mode and rebuild `main.js` on changes.
- `npm run build`: run TypeScript checks (`tsc -noEmit`) and produce a production bundle.

Example local loop:
```bash
npm run dev
```
Then reload the plugin in Obsidian to verify behavior.

## Coding Style & Naming Conventions
- Language: TypeScript with `strict` and `noImplicitAny` enabled (`tsconfig.json`).
- Indentation: 4 spaces (match existing files).
- Filenames: kebab-case (for example, `time-resolver.ts`).
- Types/classes/interfaces: PascalCase; variables/functions: camelCase.
- Prefer small, focused classes and keep side effects explicit at call sites.

No formatter script is configured; keep style consistent with existing code before submitting.

## Testing Guidelines
There is currently no automated test framework configured.

- Validate changes by running `npm run build` and ensuring no type or bundle errors.
- Perform manual checks in Obsidian: scan candidates, preview destinations, and execute archive operations.
- For bug fixes, include a short reproducible scenario and expected result in the PR description.

## Commit & Pull Request Guidelines
Git history does not yet establish a commit convention, so use clear, scoped messages.

- Recommended format: `type(scope): summary` (for example, `fix(ui): handle empty archive list`).
- Keep commits focused and avoid mixing refactors with behavior changes.
- PRs should include: purpose, key changes, manual test steps, and screenshots/GIFs for UI updates.
- Link related issues when applicable and note any settings or migration impact.
